\section*{Project Summary}

This project investigates an application-derived approach to dynamic voltage and frequency scaling (DVFS) for GPU-based high-performance computing workloads. In contrast to traditional system-level DVFS mechanisms, which rely on indirect hardware signals such as power consumption, temperature, or utilization, this work proposes a runtime control framework that derives optimization signals directly from the application itself. The core idea is to expose application-level execution semantics—such as training loop structure, kernel execution characteristics, and runtime performance indicators—through a Python decorator-based instrumentation layer. This layer enables fine-grained monitoring and control of GPU power states during execution of PyTorch-based deep learning workloads. Based on these signals, a heuristic control policy dynamically adjusts GPU frequency and power settings using available runtime interfaces (e.g., NVML and system-level power management APIs). The system is evaluated in terms of energy consumption, performance overhead, and training efficiency compared to conventional GPU DVFS governors. The goal of this work is to assess whether application-aware runtime information can improve energy-performance trade-offs in GPU computing and to explore the feasibility of integrating such control mechanisms directly into scientific software stacks.

\section{Introduction}

Since the explosion of AI, many hyperscalers, as well as research institutions, heavily invest in GPU based research and model training. Hardware manufacturers thus lean very heavily into broad vector and matrix centric calculation accelerators, most notably Graphics-Processing-Units. Given that the market demand mostly focuses on maximum performance per dollar, many other key factors are not considered, such as energy efficiency. However, not all calculations need to be run at the best performance; they can also be run at the most optimal balance between performance and energy cost. The ever growing cost and demand for energy propose the necessity to also optimise for said factors. The goal of this work is to create a Python based framework that implements different energy optimisation algorithms and uses them to adjust the GPUs Voltage and frequency settings to create a proper balance. The idea is to optimise for a key that, as opposed to traditional methods, gathers its information from the application itself. This means that optimisation keys can be determined by the developer of the scientific application and thus are easily changeable during runtime as well as throughout the development cycle.
\section{Related Work}
\subsection{COUNTDOWN: a Run-time Library for Performance-Neutral Energy Saving in MPI Applications}
The COUNTDOWN research group targets energy savings primarily through communication and synchronisation in the MPI framework. Its core idea is to improve classical static DVFS by implementing a runtime that identifies certain communication patterns. This work is very interesting for this project since it follows the same runtime driven approach. However, it is only partially application informed because it still tries to derive computationally intensive regions by itself.
\subsection{A Data-Driven Frequency Scaling Approach for Deadline-aware Energy Efficient Scheduling on Graphics Processing Units}
This work advises a classification of different energy optimisation approaches. It differentiates between analytical models that, according to the authors, use the mathematical relationships between certain components and variables in a given system. Secondly, they propose static models that use metrics like source code evaluation and hardware definitions for energy profiling. Lastly, machine learning approaches are mentioned. They use a set of input variables and try to model energy profiles as well as runtime as closely as possible based on a set of training data. This poses a different set of problems since it is hard to guarantee that novel algorithms will indeed work similarly, or at least in some aspects similar to the algorithms used during training.
\subsection{A smart energy and reliability aware scheduling algorithm for workflow execution in DVFS-enabled cloud enviornment}
This work presents a very complex variant of DVFS that enables energy optimisation in a cloud environment consisting of multiple Virtual Machines that run on a single host machine. It also incorporates a deadline based algorithm to guarantee that the workload will be finished by a given time point for the user. This opens up a lot of room for optimisation since it allows for overlapping workloads and a meta workload analysis in the sense that an independent scheduling order is possible.
\subsection{Methodology for GPU Frequency Switching Latency Measurement}
This work focuses on the latency that is needed to complete a frequency shifting cycle. They implement an array of experiments in order to achieve an accurate estimation of the underlying roofline value that is needed for frequency shifting. Also, it incorporates a similar model to this project since it uses the neighbouring CPU as the control unit. Additionally, the knowledge about said frequency changing latencies can be incorporated into later architectural decisions.
\subsection{Application-aware Memory System for Fair and Efficient Execution of Concurrent GPGPU Applications}
This joint work of some Nvidia and Intel engineers is rather peripherally interesting, but nevertheless still shows some correlation to this project. Their main goal is not only to allow multiple kernels to run in parallel but also to allow multiple applications to run concurrently. This poses some problems, as it now needs to be determined from the viewpoint of the runtime where the corresponding memory actually lies. The efficiency is consequently calculated by the overall throughput that is achieved.
\section{Theoretical Framework}

The underlying problem is a classical mathematical optimisation problem. Only a few parameters are needed to decide on the next iteration steps to change the GPU parameters since the key comes from the application itself. A simple example would be:
\begin{itemize}
    \item $f_t \in \mathbb{R}$: GPU frequency (MHz) at time step $t$
    \item $k \in \{\text{key}, \text{runtime}\}$: optimisation metric
    \item $R(f_t, k)$: performance evaluation function (higher is better)
    \item $t \in \mathbb{N}$: iteration index of the optimisation process
\end{itemize}

\textbf{Objective:}

\[
f^{*} = \arg\max_f \; R(f, k)
\]

\textbf{Update rule (discrete hill-climbing):}

\[
f_{t+1} =
\begin{cases}
f_t + 50 & \text{if } R(f_t, k) > R(f_{t-1}, k) \\
f_t - 50 & \text{otherwise}
\end{cases}
\]
Here, a simple step function is used, which brings some problems by itself, such as a high amount of jitter around the local minimum. Later in this work, a new algorithm will be proposed that incorporates a profiling mechanism to scan a GPU in regard to a specific algorithm and then decide on an energy optimal frequency and voltage setting.
\section{Traditional Approach and Key Innovations}
Traditionally, DVFS systems rely on the input variables of the current power draw, temperature, and resource utilisation. This introduces the problem that these variables sometimes do not directly influence the actual problem as strongly as they scale up the power draw. This highlights the need for a better heuristic that is directly linked to the actual application performance. Since this is only possible by explicitly stating these variables during runtime, it needs to be incorporated by the developer. The key innovation of this work is a novel optimisation algorithm that uses a combination of traditional optimisers, such as gradient descent, and attempts to measure its performance in comparison to other optimisation functions. The key difference, though, is that the application informed heuristic allows for much closer and better optimisation of the actual model training.
\section{Research Gap}
The biggest research gap is the application blindness. Most other approaches focus mainly on a system level and do not further incorporate the application layer. This systems level signal is very indirect and leaves the actual algorithm as a black box. As a result, the implemented governors are fairly detached from the actual runtime. Works like the COUNTDOWN show that a dynamic approach with application is possible while they still try to infer optimisation options via communication patterns rather than information from the application itself.\newline
Another key aspect is the target user group. Almost every related work focuses on large-scale applications and hyperscalers, given that they represent the most profitable market, of course. However, small Research Models for experimentation and research should also be taken into account, since there is plenty of room for optimisation, and decentralised applications still make up a huge portion of energy consumption.
\newline
The third tangent that shows no real evaluation is the impact of different levels of granularity for energy optimisation calls in a training context. It needs to be determined whether to change the DVFS values based on a batch level or an epoch level. This is very interesting for deep learning applications since these tend to run for a long time and thus offer a lot of room for frequency and voltage changes. 

\section{Possible Research Question}
 \begin{itemize}
     \item To what extent can developer-defined, application-layer performance metrics serve as effective control signals for GPU DVFS, and how do the resulting energy-performance trade-offs compare to system-level DVFS governors?
     \item Does DVFS control granularity — batch-level versus epoch-level adjustment — significantly affect training throughput stability and net energy savings in GPU-based deep learning workloads?
     \item What is the instrumentation and control overhead introduced by the GreenTorch framework, and under what conditions does it outweigh the energy savings achieved?
     \item How sensitive is the proposed heuristic to the choice of application-defined optimisation key, and what properties must a metric satisfy to yield stable and meaningful DVFS control?
 \end{itemize}

\section{Methods}
The main methodology will be the implementation and evaluation of different energy saving optimisers. The research field in that regard is relatively sparse, meaning there is a lot of room for improvement. In addition to that, most research focuses on large-scale applications, whereas this work is more targeted toward the small, research-focused models used in most biology, geo-science, physics, etc.
Another unique feature of this project will be the ease of use in integrating energy optimisation in a very simple manner:
\begin{minted}{python}
def train():
    with GreenTorch() as gt:
        while training_loop:
            # Training steps
            gt.key = 1 / time_diff
            gt.optimise()
\end{minted}
The context manager initiates a background analysis thread that continuously monitors GPU energy consumption and runtime performance metrics.
Application-specific optimisation signals are communicated to GreenTorch through a user-defined key. This key represents a runtime metric that should be maximised and must be closely related to the application's performance objective. Suitable examples include training throughput metrics such as batches per second or samples per second.
To ensure meaningful optimisation, the selected metric should be measurable at runtime, available at fine-grained intervals, and robust against short-term fluctuations.
For prototyping purposes, the framework utilises the LACT project to access GPU telemetry and modify GPU operating parameters, including frequency and, where supported, voltage settings. LACT provides a hardware abstraction layer implemented in Rust that exposes a unified interface to underlying GPU driver stacks while offering access to hardware metadata and control functionality.
The current implementation focuses on a single GPU backend to ensure reproducibility and limit system complexity. Support for additional vendor-specific interfaces, such as native AMD and NVIDIA driver APIs, is considered future work.
